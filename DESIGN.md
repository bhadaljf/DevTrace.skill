# DevTrace Skill 设计文档

## 1. 定位

DevTrace 是一个围绕跨时间事件追踪工作的记录系统。

它的目标有四个：

- 把项目推进中的关键事件沉淀成可持续追踪的事实单元
- 让 AI 在后续 session 中更快接上上下文
- 让用户能直接看到最近生成了什么记录内容
- 让日报、周报、月报、review、总结都建立在同一套事件事实之上

DevTrace 的核心不是聊天切块，也不是一次性总结；核心是让同一个事件在跨时间推进中始终有一条连续可追的记录。

---

## 2. 核心问题

DevTrace 主要解决这些问题：

1. 新 session 进入项目时，AI 很难快速接上当前状态
2. 项目推进信息散落在聊天、会议、实验、调试、复盘里
3. 用户和 AI 都缺少一个统一入口来查看最近记录了什么
4. 重要进展缺少跨时间的事件事实沉淀
5. 日报、周报、月报、review 缺少稳定的事实来源

所以 DevTrace 不是在堆总结，而是在建立一套稳定的：

- 当前状态入口
- 生成内容入口
- 事件事实层

---

## 3. 结构总览

DevTrace 现在应明确分成四层：

1. **当前状态层**
2. **生成内容索引层**
3. **事件事实层**
4. **筛选与关系辅助层**

其中核心对象与职责固定如下：

- `01_CURRENT.md`：当前状态入口
- `00_INDEX.md`：生成内容索引入口
- `TraceUnit`：事件事实层
- `tag / thread / date / status / affects_current`：筛选辅助
- `prev / next / related / supersedes`：关系辅助

这里最关键的变化是：

> `00_INDEX.md` 不再定义为 TraceUnit 索引；它定义为生成内容索引。  
> `TraceUnit` 继续作为事实层存在，主要供 AI 在需要时深入读取。

---

## 4. 核心对象

### 4.1 `01_CURRENT.md`

`01_CURRENT.md` 是当前状态入口。

它回答的是：

- 当前主线是什么
- 当前问题是什么
- 当前 blocker 是什么
- 下一步是什么
- 最近变化是什么

它的作用是：

- 让 AI 先进入“现在”
- 给后续读取提供最短路径

它不承担完整历史，不代替事件事实层。

### 4.2 `00_INDEX.md`

`00_INDEX.md` 是生成内容索引。

它面向：

- 用户
- AI

它记录的应该是：

- 用户主动让 AI 生成的内容
- 定时自动生成的内容
- 可供后续回看与导航的内容入口

典型内容包括：

- 当前会话总结
- 阶段总结
- review
- 日报
- 周报
- 月报
- 定时生成的日报 / 周报 / 月报

它的作用是：

- 让用户看最近生成了什么
- 让 AI 看最近记录了什么
- 让 AI 根据 tag、类型、时间线决定后续往哪里读

它不承担事件细节，不代替 `TraceUnit`。

### 4.3 `TraceUnit`

`TraceUnit` 是事件事实层，也是长期维护的主对象。

它记录的是：

- 值得跨时间继续追踪的事件
- 同一事件在不同时间里的推进细节
- 后续总结、review、日报周报的事实来源

它的作用是：

- 给 AI 保留长期事件记忆
- 让 AI 在需要时继续深入追踪事件

`TraceUnit` 默认按一个大事件沉淀；同一事件里的 `problem / attempt / decision / next-step / result` 很多时候只是 unit 内部的一部分，只有在形成新的独立状态节点、值得未来单独加载时，才更适合单独成为新的 unit。

### 4.4 `tag`

`tag` 是主题标签。

它可以出现在：

- `TraceUnit`
- `00_INDEX.md` 的记录项

它的作用是：

- 帮助用户快速定位主题
- 帮助 AI 从 index 找相关生成内容
- 帮助 AI 从 unit 找同主题事件

`tag` 的主体来自用户和 AI 的实际判断，脚本只应做轻量辅助。

---

## 5. `TraceUnit` 设计

### 5.1 基本判断

判断一段内容是否值得成为 `TraceUnit`，核心只看一件事：

> 后面还会不会沿着这件事继续追。

如果答案是“会”，它通常值得成为 `TraceUnit`；如果只是同一事件里的补充细节，它通常更适合续写已有 unit。

### 5.2 默认粒度

当前阶段默认粒度是：

> 一轮当前会话，优先沉淀为一个大事件 unit。

只有本轮真的出现第二个独立大事件，才拆多个 unit。

### 5.3 新建与续写

写回前最重要的判断不是“切几个 unit”，而是：

1. 这轮是不是值得沉淀的大事件
2. 这是新事件
3. 还是旧事件续写

默认原则：

- 能续写旧事件，就不要默认新开 unit
- 只有形成独立新状态时，才新开 unit

### 5.4 独立新状态

这里的关键不是“内容是否相关”，而是：

> 这轮进展是否已经成为未来值得单独加载的一次状态转折。

典型情况：

1. 旧 blocker / 旧问题还在，只是补充细节、补做验证、继续排查
   - 优先续写旧 unit

2. 旧 blocker 还在，但已经形成明确 next-step / 决策
   - 更适合新开 unit

3. 旧 blocker 被解除，或者拿到了明确结果
   - 更适合新开 `result` unit

4. 主线、方案、理解口径发生转向
   - 更适合新开 `decision / direction-shift` unit

### 5.5 关系字段

关系字段只保留最小导航：

- 前后承接：`prev / next`
- 横向强相关：`related`
- 新状态覆盖旧状态：`supersedes`

它们只在关系已经明显、并且值得未来继续追踪时再写；默认不追求把所有 unit 串成复杂事件图。

---

## 6. `00_INDEX.md` 设计

### 6.1 定位

`00_INDEX.md` 记录最近生成了什么内容，而不是记录所有 `TraceUnit`。

它的核心作用是：

- 给用户一个最近记录内容的浏览入口
- 给 AI 一个最近生成内容的导航入口

### 6.2 写入范围

应该写进 `00_INDEX.md` 的内容：

- 当前会话总结
- 阶段总结
- review
- 日报
- 周报
- 月报
- 定时生成内容
- 其他明确生成的记录文档

默认不直接写进 `00_INDEX.md` 的内容：

- 单纯新建 `TraceUnit`
- 单纯续写 `TraceUnit`
- 只更新事实层、没有生成新的导航内容

### 6.3 单条格式

```md
- YYYY-MM-DD [path/to/file.md] 一句话概括。 #tag1 #tag2 #type
```

例如：

```md
- 2026-04-04 [reviews/daily/2026-04-04.md] 今日日报：读取顺序已经固定，主线转向筛选规则细化。 #loader #design #daily
- 2026-04-04 [reviews/weekly/2026-W14.md] 本周回顾：完成 current -> index -> 相关内容 的默认读取路径收敛。 #loader #review #weekly
- 2026-04-03 [reviews/session/2026-04-03-loader-summary.md] 会话总结：确认 index 作为生成内容入口，unit 作为事件事实层。 #loader #design #session-summary
```

每条记录最少包含：

- 日期（`YYYY-MM-DD`）
- 文件入口
- 一句话概括
- tag

### 6.4 年月结构

`00_INDEX.md` 按年、月组织。

文件开头先放年、月跳转，例如：

```md
# 00_INDEX

- [2026年](#2026年)
  - [2026-04](#2026-04)
  - [2026-05](#2026-05)
- [2027年](#2027年)
```

正文组织：

```md
## 2026年

### 2026-04

- 2026-04-04 [reviews/daily/2026-04-04.md] 今日日报：读取顺序已经固定。 #loader #daily
- 2026-04-03 [reviews/session/2026-04-03-loader-summary.md] 会话总结：确认新的读取路径。 #loader #session-summary

### 2026-05

- 2026-05-01 [reviews/monthly/2026-05.md] 月报：本月主线进入筛选规则细化。 #loader #monthly
```

要求：

- 每年使用单独标题，例如 `## 2026年`
- 每个月使用单独标题，例如 `### 2026-04`
- 每个月之间空一行

### 6.5 类型约定

固定几类常见记录类型，用 tag 表示：

- `#session-summary`
- `#review`
- `#daily`
- `#weekly`
- `#monthly`

这样用户和 AI 都能快速识别记录类型。

## 7. AI 阅读流程

AI 默认按这个顺序工作：

1. 先读 `01_CURRENT.md`
2. 再读 `00_INDEX.md`
3. 根据 index 中的：
   - 时间线
   - tag
   - 记录类型
   - 一句话概括
4. 决定后续要读：
   - 哪份日报 / 周报 / review / 总结
   - 哪些相关 `TraceUnit`

也就是说：

- `current` 管现在
- `index` 管最近生成了什么
- `TraceUnit` 管事件事实

这里最关键的一点是：

> `TraceUnit` 是事实层，不是默认第一入口；AI 先从 current 和 index 进入，再按需下钻到相关 unit。

---

## 8. 写回流程

正确写回流程应是：

1. AI 理解当前会话或当前任务
2. 判断是否需要生成新的导航内容
3. 判断相关事件是新建还是续写 `TraceUnit`
4. 写入或续写 `TraceUnit`
5. 如果本轮产生了新的总结 / review / 日报周报等生成内容，再写入 `00_INDEX.md`
6. 按需要更新 `01_CURRENT.md`

脚本负责：

- 稳定落盘
- 稳定更新 current / index
- 提供读写入口

AI 负责：

- 事件理解
- 粒度判断
- 新建 / 续写判断
- 生成内容判断
- tag 判断
- relevant 内容的最终阅读判断

---

## 9. 视图层

日报、周报、月报、review、会话总结都属于视图层产物。

它们的定位是：

- 面向用户和 AI 的回顾内容
- 面向 index 的导航内容
- 面向 `TraceUnit` 的上层组织视图

它们不替代事实层，只建立在事实层之上。

---

## 10. 定时自动化

DevTrace 的定时生成只保留一种方案：

> 用户用自然语言告诉 AI 什么时候生成日报 / 周报 / 月报，由 AI 自己创建并维护自动任务。

不再把本地调度器、外部计划任务作为默认方案。

### 10.1 用户入口

用户可以直接说：

- 以后每天晚上 9 点生成日报
- 每周五下午生成周报
- 定时生成月报

AI 需要把这些话理解成：

- 生成什么内容
- 什么时候生成
- 写到哪个项目目录

### 10.2 自动任务的职责

自动任务运行时应完成：

1. 读取 DevTrace 当前项目上下文
2. 生成对应周期的 review
3. 写入 `reviews/` 下对应目录
4. 把生成结果写入 `00_INDEX.md`

典型输出目录：

- `reviews/daily/`
- `reviews/weekly/`
- `reviews/monthly/`

### 10.3 与现有结构的关系

定时日报 / 周报 / 月报属于视图层产物：

- 事实来源仍然是 `TraceUnit`
- 当前状态入口仍然是 `01_CURRENT.md`
- 导航入口仍然是 `00_INDEX.md`

也就是说，自动化不会引入新的核心对象；它只是把现有 `generate-review` 能力改成可重复执行的 AI 任务。

### 10.4 自动化写回

定时任务生成完成后，必须继续遵守已有写回规则：

1. 生成日报 / 周报 / 月报文件
2. 追加一条新的 index 记录
3. 不因为生成 review 而直接改写 `TraceUnit`
4. 只有确实需要时才更新 `01_CURRENT.md`

### 10.5 月报的处理

如果自动化平台不能稳定表达“每月一次”，也仍然只保留 AI 自动任务方案。

处理方式是：

- 由 AI 创建一个更高频的自动任务
- 在任务内部判断当前是否到了该生成月报的时间
- 只有满足条件时才真正生成月报

这样仍然保持：

- 用户只和 AI 说一句话
- AI 自己处理后续定时逻辑
- DevTrace 内部结构不需要改变

## 11. 一句话总结

> DevTrace 的稳定结构是：`01_CURRENT.md` 管当前状态，`00_INDEX.md` 管生成内容入口，`TraceUnit` 管事件事实，AI 先看 current，再看 index，再按 index 和 tag 去读相关生成内容或事件单元；定时日报 / 周报 / 月报也由 AI 自动任务生成，并继续写回 reviews 与 index。
