# Workflow

DevTrace 当前阶段的稳定工作流应按下面四层结构理解：

```text
当前状态层(current)
-> 生成内容索引层(index)
-> 事件事实层(TraceUnit)
-> 筛选与关系辅助层(tag/thread/date/status/affects_current + relations)
```

## 1. 先读当前状态

优先读取：

- `01_CURRENT.md`

作用：

- 先知道项目现在到哪了
- 先知道当前最该关注什么

## 2. 再读生成内容索引

再读取：

- `00_INDEX.md`

作用：

- 看最近生成了什么内容
- 看时间线
- 看 tag
- 看有哪些记录值得继续读

## 3. 再决定后续读取方向

不要直接把所有 `TraceUnit` 全读一遍。

先根据任务决定接下来是：

- 继续读某份会话总结 / review / 日报 / 周报 / 月报
- 还是进一步进入相关 `TraceUnit`

作用：

- 让 index 先承担导航
- 让事实层只在需要时再被读取

## 4. 再进入事实层

事实层只有：

- `TraceUnit`

当 current 和 index 已经不足以支持当前任务时，再进入相关 `TraceUnit` 去看：

- 事件具体发生了什么
- 当前状态是怎么形成的
- 哪些细节值得继续追踪

## 5. 读取的执行顺序

### 场景 A：继续当前主线

1. 读 current
2. 读 index
3. 优先看最近的相关总结 / 阶段记录
4. 如仍不够，再读相关 unit

### 场景 B：追某个具体事件

1. 先从 index 找到相关记录
2. 再看同 tag / 同时间线内容
3. 如仍不够，再读相关 unit
4. 必要时沿关系字段继续追

### 场景 C：查看某个主题

1. 先按 index 中的 tag 找相关内容
2. 再看最近时间
3. 如仍不够，再补读同主题 unit

### 场景 D：做 review / 阶段复盘

1. 先看时间窗内的日报 / 周报 / 月报 / review
2. 再看当前主线相关内容
3. 必要时再补读高价值 unit

## 6. 什么时候继续扩读

默认不要读太多 unit。

只有这些情况才扩读：

- current 和 index 还不足以解释当前状态
- 用户要看某个事件全貌
- 当前任务本身就是追某个事件的来龙去脉
- 当前生成内容需要回到底层事实补细节

## 7. 当前会话写回

当前会话写回顺序：

1. AI 理解当前会话
2. 判断追踪对象
3. 判断这是新事件还是旧事件续写
4. 写入或续写 `TraceUnit`
5. 如果本轮还生成了新的总结 / review / 日报周报，再更新 `00_INDEX.md`
6. 按需更新 `01_CURRENT.md`

## 8. review / 阶段回顾

review 的组织顺序应优先是：

1. 当前状态背景
2. 本周期关键推进
3. 关键决策与转向
4. 风险与 blocker
5. 下一步建议
6. 参考 TraceUnit

review 是建立在事实层之上的输出视图，不替代事实层本身。

## 9. 旧事件续写

如果本轮只是已有事件的继续推进，优先：

- 定位旧 `TraceUnit`
- 追加 `details / evidence`
- 按需改 `status / summary / relation`

但如果当前轮已经形成了新的状态节点，就不要机械续写：

- 旧 blocker 还在，但已经形成明确 next-step / 决策：优先新开 unit
- 旧 blocker 被解除，形成明确结果：优先新开 `result` unit
- 主线或方案发生转向：优先新开 `decision / direction-shift` unit

## 10. 关系字段怎么轻量使用

关系字段只做最小导航：

- 明显前后承接：写 `prev / next`
- 明显横向强相关：写 `related`
- 明显覆盖旧状态：写 `supersedes`

不要为了“图完整”而强行补关系。
